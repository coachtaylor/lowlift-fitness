35 self-contained prompts — one per movement. Each block already includes the brand style,

## ✅ PRODUCTION RECIPE (LOCKED — read this first)

The bodyweight-squat is built and approved in-app. Match it exactly for all 34 others so the
set is consistent. **The biggest risk is character drift** — keep it the same person throughout.

**Per frame set, every movement:**
1. **Same character.** Reuse the exact figure from the approved squat (same face, hair, volt-green
   tank, light-grey joggers, neutral mid-tone skin). Lock a **seed or reference image** of a clean
   standing frame and reuse it for all 34. Mismatched characters = looks like a grab-bag. ← #1 rule.
2. **Generate BIG, downscale in post.** Output a **3072×3072 px** sprite sheet (4×4 = 16 tiles,
   each **768×768 px**); 2048×2048 / 512 px tiles is the acceptable minimum. The final webp's pixel
   size doesn't matter for on-screen size (the player scales every clip into the same fixed stage with
   `contentFit:contain`) — so I downscale to one uniform canvas in post. **Keep the hi-res originals**
   (they're the source for the final quality re-do pass — no regenerating). Earlier batches came in
   small (~283–384 px) because the tool defaulted low; the big-sheet approach fixes that.
2b. **FIXED character bounding box** in every tile: same character scale and position in all 16 tiles
   (~70% of the tile), feet/lowest point on the SAME floor line, body's vertical center on the same
   vertical line — only the moving part changes frame to frame. This is what keeps movements consistent
   and minimizes post-stabilization. **🔒 The character must NOT slide, translate, or reposition between
   tiles** (no drifting left/right or up/down). Pin the body and feet; animate ONLY the moving part
   (head tilt, arm reach, knee lift). *Observed failure: a generator "animated" a neck stretch by
   sliding the whole upright figure sideways across the tiles instead of tilting the head — the result
   is useless. The motion must be a real change of pose, not a change of position.*
3. **16 frames / 16 tiles.** Either a single 4×4 sprite-sheet PNG (I slice it) or 16 PNGs named
   `frame_01.png … frame_16.png` — either works.
4. **Background:** solid off-white `#F7F7F5`. Figure centered with **margin** (no edge cutoff),
   **no stray elements** (the first squat had floating shoes — the negative prompt covers this).
5. **Seamless loop:** the motion should return to its start so frame 16 → frame 01 is smooth.

**⏱ SPEED IS UNIFORM — everything plays at the bodyweight-squat tempo: `240 ms/frame`.**
A 16-frame loop = **3.84 s**. There is no per-type speed anymore; the on-screen motion speed is the
same for every movement. (Set at **encode** time — don't worry about frame timing while generating;
just make a clean 16-frame loop.)

**Loop motion still differs by movement type (the *kind* of motion, not the speed):**
| Type | Movements | Motion | Frames |
|---|---|---|---|
| **Reps** | squats, push-up, calf raise, glute bridge, sit-to-stand, arm circles, heel-to-toe | full down→up rep cycle | 16 |
| **Holds** | plank, bear plank, wall sit, wall push | **Hold the position.** Most encode as a **static single-frame freeze** (no loop, no bobbing). If the gen includes a real entry (e.g. wall push leaning in), I play the entry once then freeze the last frame. | 16 (entry) → freeze |
| **Stretches** | neck, chest opener, hamstring, hip flexor, quad, shoulder, side body, wrist, figure-four, child's pose, forward fold, spinal twist, cat-cow, ankle circles, shoulder rolls | **deepen-and-release cycle** spread across ALL 16 frames (ease in → deepest 1–2f → ease out → loop). NEVER "ease-in then hold" — that stalls. Per-side ones generate one side (I mirror). | 16 |

**Alternating/bilateral movements** (reverse lunges, side leg raises, side-step, knee extensions,
**knee drive, dead bug**, marches, cross punches) get a **full 16 frames per side** → a ~32-frame loop,
so each side's rep still runs at the exact squat tempo. Generate one clean side and I mirror it, or
generate both sides natively. ⚠️ **Mirror (`-flop`) only works cleanly for front/near-front standing
views** — a *lying/top-down* pose (like dead bug) flips head L↔R when flopped, so prefer a **native
both-sides** gen for those (16f diagonal A → 16f diagonal B, head orientation unchanged).

**📦 Handoff convention:** deliver each movement named exactly by its **`<slug>`** (slugs are in the
mapping table at the bottom) — either the **single 3072² sprite-sheet PNG** (`<slug>.png`, I slice the
16 tiles) or a **zip of the 16 sliced frames** (`<slug>.zip` of `frame_01.png … frame_16.png`). Either
works. Drop them on me **as you finish — batches of ~5 is great** — and I'll encode incrementally.

**Frames vs GIF:** send **frames** (PNGs) when you can — full color, sharper, and they let me set
the per-type loop speed at encode time. A **GIF** is an accepted fallback if it's much less hassle
(I'll extract + re-encode it), but GIF is 256-color so the art loses a little quality. If GIF, name
it `<slug>.gif`.

## STRETCHES — Static

> **Encode convention for stretches — generate accordingly:**
> - **⛔ ANTI-STALL (the #1 rule for every stretch):** the motion must be spread **evenly across all 16 tiles** so each tile is a visibly different step. The most common failure is the generator reaching the final pose in the first ~4 frames and then repeating a near-identical static pose for the rest — **do not do this.** Frame the motion as a slow **deepen-and-release cycle**: ease from a neutral start into the deepest stretch over roughly the first half, hold the deepest point for only 1–2 frames, then ease back out over the second half so frame 16 flows back into frame 1 (a seamless loop with visible movement throughout).
> - **⛔ ANTI-SLIDE (just as important):** the change between frames must be a real **change of POSE** (the head tilts, the arm reaches, the knee lifts), NOT a change of POSITION. The character's body and feet stay pinned in the exact same spot — it must never slide, drift, or reposition across the tiles. *(A neck-stretch sheet failed by sliding the whole upright figure sideways instead of tilting the head — unusable.)*
> - **Per-side** (hamstring, hip-flexor, quad, neck, shoulder, side-body, wrist, figure-four, spinal-twist): run the deepen-and-release cycle on **ONE side only**, centered and facing the camera; do NOT switch sides mid-clip (the other side is **mirrored** in post).
> - **Symmetric static** (chest-opener, child's-pose, forward-fold): same deepen-and-release cycle, both sides of the body together.
> - **Dynamic** (ankle-circles, cat-cow, shoulder-rolls): a smooth **seamless cyclic loop** — frame 16 must flow continuously back into frame 1.
> Keep everything gentle — **no bouncing, no straining, no jerk** — but always **moving** (no dead, frozen, repeated frames).
